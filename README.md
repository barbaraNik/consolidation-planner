# ConsolidationPlanner

Сервис консолидации грузов и расчёта загрузки морского контейнера с целью оптимизации процесса закупок.

## О проекте
Техническое задание бизнес аналитика для веб-сервиса,
автоматизирующего подбор заказов в морской контейнер с расчётомоптимальной загрузки.
## 📊 Бизнес-контекст проекта

### Проблематика

В рамках анализа бизнес-процессов департамента закупок компании по импорту промышленного оборудования из Китая было выявлено **критическое узкое место**:

**Проблема:**  
Ручной подбор заказов для консолидации в морские контейнеры занимал у менеджеров **до 2 часов** на одну задачу.
Процесс включал:
- Выгрузку данных о заказах из 1С:ERP 
- Ручной расчёт габаритов и весов по каждому заказу
- Подбор комбинаций заказов методом перебора
- Проверку соответствия ограничениям контейнера (объём, грузоподъёмность)
- Согласование итогового списка с отделом закупок
- Согласование итогового набора заказов споставщиком

**Последствия:**
- **Высокая трудоёмкость:** до 2 часов/задачу на рутинные расчёты
- **Финансовые потери:** неоптимальная загрузка контейнеров 
- **Ошибки:** периодические нарушения ограничений по весу, задержки отгрузок
- **Отсутствие метрик:** невозможность объективно оценить качество консолидации

### Предложение для компании:

**Цель проекта:**  
Разработать веб-сервис автоматизации консолидации грузов для **снижения затрат на логистику** и **высвобождения ~87% рабочего времени менеджеров** для других задач.

**Ожидаемые результаты:**
- Сокращение времени подбора заказов с 2 часов до 15 минут 
- Повышение коэффициента загрузки контейнеров 
- Снижение количества ошибок при формировании консолидаций до 0
- Внедрение объективных метрик качества (PV - загрузка по объему, PM - загрузка по массе, F-score)
- Возможность аудита и анализа истории консолидаций

**Более того это поспособствует:**
- Экономии на недозагруженных контейнерах
- Экономии трудозатрат менеджеров и повышении их эффективности
- Снижение затрат на срочные доставки из-за ошибок

## Роль в проекте

**Должность:** Бизнес аналитик

**Выполненные работы:**

### Анализ текущего процесса (AS-IS)
- Провела интервью с членами департамента и руководителем отдела логистики
- Замерила фактическое время выполнения операций 
- Выявила узкие места: ручной расчёт габаритов, отсутствие валидации ограничений
- Построила диаграмму процесса в нотации BPMN
- Зафиксировала текущие метрики: средний коэффициент загрузки PV = 0.58

### Проектирование целевого процесса (TO-BE)
- Разработала концепцию автоматизированного подбора с алгоритмом оптимизации
- Описала пользовательские сценарии (User Stories)
- Согласовала бизнес-правила с отделом закупок и логистики
- Утвердила критерии приёмки с системным аналитиком и отделом ИТ

### Проектирование системы
- **Модель данных:** спроектировала структуру БД (PostgreSQL)
- **API-спецификация:** REST API-методы согласно требованиям
- **Алгоритмы:** описала логику автоподбора заказов и расчёта метрик загрузки 

### Документация
- Техническое задание (бизнес-цели, ФТ, НФТ, критерии приемки) 
- API-спецификация
- ERD-диаграмма структуры базы данных
- DDL-скрипт создания схемы БД (PostgreSQL)

**Метрики качества консолидации:**
```
F-score = 0.6 × PV + 0.3 × PM + 0.1 × PN

где:
  PV — коэффициент загрузки по объёму 
  PM — коэффициент загрузки по весу 
  PN — коэффициент загрузки по количеству заказов 
```

**Интерпретация:**
- F ≥ 0.80 — отличная консолидация (зелёный индикатор в UI)
- F ∈ [0.50, 0.80) — удовлетворительная (жёлтый)
- F < 0.50 — низкая загрузка, рекомендуется добавить заказы (красный)
