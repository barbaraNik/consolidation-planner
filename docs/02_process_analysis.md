### AS-IS и TO-BE процессы (BPMN)

## 1. Контекст анализа

В ходе анализа всего процесса закупки и логистики было выявлено **критическое узкое место** на этапе консолидации грузов в морские контейнеры.

Данный раздел посвящён **детальному анализу этого узкого места** с использованием нотации BPMN.

## 2. Текущий процесс консолидации (AS-IS)

### 2.1. Обзор процесса
1. **Инициация консолидации**: менеджер принимает решение, можно ли объединить заказ с другими
2. **Выгрузка данных из 1С**: вручную формируется список заказов со статусом «Груз готов» для конкретного порта
3. **Расчёт габаритов и весов**: для каждого заказа менеджер вручную суммирует объёмы и веса всех грузовых мест (коробов)
4. **Подбор консолидации**: методом перебора подбираются комбинации заказов, которые укладываются в ограничения одного из контейнеров (если умещается в 40DC, то 40DC выбирается для поставки, если много пустого просранства - рассматривается вариант поставки в меньшем контейнере)
5. **Проверка ограничений**: вручную сверяется суммарный объём и вес с характеристиками контейнера
6. **Согласование с поставщиком**: менеджер отправляет предложенный набор заказов поставщику для подтверждения возможности загрузки
7. **Цикл уточнений** (поставщик может сообщить):
   - "Груз не уместится": менеджер исключает часть заказов и повторяет шаги 4-6
   - "Контейнер недозагружен": менеджер добавляет заказы и повторяет шаги 4-6
   - "Согласовано" : консолидация утверждается
8. **Передача в логистику**: готовая консолидация передаётся для организации отгрузки

### 2.2. Узкое место процесса

**Узкое место:**  
Ручной подбор заказов (шаг 4) и итерационное согласование с поставщиком (шаги 6-7).

**Проблемы:**

| Проблема | Последствия |
|----------|-------------|
| **Ручной расчёт габаритов** | Ошибки в расчётах может привести к нарушению ограничений контейнера, что приведет к срыву сроков отгрузки |
| **Метод перебора без алгоритма** | Неоптимальные комбинации может выявить низкую загрузку контейнера (PV = 0.58 вместо 0.75+) |
| **Итерации согласования** | 2-3 цикла уточнений с поставщиком приводит к задержкам на 1-2 дня (что, в свою очередь, может тормозить весь цикл поставки на более, чем 5 дней |

## 3. Целевой процесс консолидации (TO-BE)

### 3.1. Как должно быть

1. **Инициация консолидации**: cистема автоматически отслеживает готовые заказы и инициирует консолидацию
2. **Выгрузка данных**: cистема автоматически загружает данные через API
2.1 **Действия менеджера:**
 2.1.1 Менеджер открывает экран "Консолидация грузов и расчёт загрузки контейнера" в системе
 2.1.2 Выбирает из выпадающего меню Порт, тип контейнера, дату или выбирает автоматический подбор грузов
 2.2.1 **Система отображает список заказов-кандидатов на загрузку**
4. **Расчёт габаритов и весов**: система автоматически рассчитывает из данных и показывает метрики для каждого заказа
5. **Подбор консолидации**: система предлагает оптимальный вариант автоматически
6. **Передача в логистику**: готовая консолидация передаётся для организации отгрузки


**Результат:**
- Сокращение времени с 2 часов до 15 минут (на ~87%)
- минимизация ошибок благодаря автоматизации 
- Наглядные метрики (PV - загрузка по объему, PM - загрузка по весу, F-score)
- Сокращение этапов формирования консолидации
- Повышение загрузки контейнеров благодаря автоматизированному просчету
